# ADR 0016: UTO Studio — Visual Test Authoring and Recording

Date: 2026-03-18

## Status

Proposed — Planned for Phase 6

## Context

With the completion of Phase 5 (`uto ui`), UTO gains an interactive browser-based interface for running, watching, and debugging test suites. The next differentiation milestone is **test creation**: removing the need to hand-write automation code for the most common interaction patterns.

### The Cypress Studio Gap

Cypress Studio is a visual test recorder built into the Cypress test runner. It allows developers to:

- Click through a running application and have Cypress generate `.cy.js` assertions.
- Interactively add `cy.get()` + `.click()` / `.type()` calls by pointing at DOM elements.
- Save new test commands back to existing test files.

Despite being a step toward usability, **Cypress Studio has critical limitations**:

| Limitation | Detail |
|---|---|
| **Web-only** | No mobile recording or cross-platform replay. |
| **Selector-brittle** | Generates CSS/XPath selectors that break on refactors or dynamic class names. |
| **Framework lock-in** | Recordings are coupled to Cypress DSL; non-portable. |
| **No visual intelligence** | Relies entirely on DOM structure, not visual context. |
| **No AI assist** | Cannot suggest test steps, infer intent, or validate recorded coverage. |
| **Deprecated risk** | Cypress has paused active development of Studio in recent versions. |

### Playwright's Codegen Gap

Playwright offers `playwright codegen` as a CLI-based recorder that watches mouse/keyboard events and generates Playwright test code. It shares many of the same limitations: selector-brittle, web-first, no vision, no mobile.

### The UTO Opportunity

UTO already has:

1. **Vision-first recognition loop** (Phase 3) — identifies UI elements visually, not by fragile selectors.
2. **Intent API** (`select`, `click_intent`, `fill_intent`) — human-readable, platform-agnostic test DSL.
3. **Interactive UI mode** (Phase 5) — a local browser-based interface for running and observing tests.
4. **Unified web + mobile session model** — same API surface for Chrome and Appium targets.

UTO Studio layers on top of this foundation to deliver a **visual test authoring experience that is categorically superior to Cypress Studio and Playwright Codegen**:

- Records interactions as **vision-first intent steps**, not brittle CSS/XPath selectors.
- Works for **both web and mobile** sessions within the same recording session.
- Uses **AI-assisted step suggestions** powered by the recognition loop and accessibility tree.
- Generates **portable, human-readable Rust test code** using the `uto-test` helper API.
- Integrates directly into the `uto ui` interface — no separate tool, no extra setup.

## Decision

UTO will deliver **UTO Studio** as its Phase 6 milestone — a visual, interactive test authoring and recording environment that sets a new standard for cross-platform test creation.

### Core Capabilities

#### 1. Live Session Recorder

The user launches `uto ui --studio` (or presses the "Record" button in the existing `uto ui` interface). A new browser or mobile session is opened. Every interaction — click, type, swipe, navigation — is captured as a **semantic intent step**:

```rust
// Generated by UTO Studio recording
session.click_intent("Add to Cart").await?;
session.fill_intent("Quantity", "2").await?;
session.click_intent("Checkout").await?;
session.assert_visible("Order Confirmed").await?;
```

No CSS selectors. No XPath. No platform-specific syntax. The recording speaks user intent.

#### 2. Vision-First Element Inspector

When hovering over the live app during recording, UTO Studio overlays:

- **Visual bounding boxes** around recognized UI elements (from the vision model).
- **Accessibility label hints** for elements that have metadata (role, name, etc.).
- **Confidence score** for each recognized element (so test authors can validate recognition quality).
- **Intent suggestion** — the most likely intent verb for the element (click / fill / assert).

This inspector is the **recording equivalent of Cypress Studio's DOM picker**, but grounded in visual recognition instead of CSS selectors.

#### 3. Cross-Platform Recording Sessions

A single UTO Studio recording session can target:

- A **web browser** (Chrome, Firefox, Safari via WebDriver).
- An **Android device or emulator** (via Appium + UiAutomator2).
- An **iOS simulator or device** (via Appium + XCUITest).

The test code generated references the same `uto-test` intent API regardless of platform. Platform-specific behavior is handled transparently by `uto-core`.

#### 4. AI-Assisted Step Completion

As the test author records, UTO Studio's AI layer (powered by the Phase 3 recognition loop and, optionally, an LLM adapter in future phases) suggests:

- The next most likely step based on current page/screen context.
- Alternative assertion targets visible in the current viewport.
- Warnings when a recorded element has low visual confidence (fragile target candidate).
- Corrections when a recorded step resolves ambiguously (multiple similar elements found).

#### 5. Test Code Generation

When recording is finished, UTO Studio:

1. Emits a fully-formed Rust test function using the `uto-test` helper API.
2. Writes the generated test to the project's `tests/` directory (or previews it in the editor panel).
3. Preserves round-trip fidelity — generated code is immediately runnable via `uto run`.
4. Formats output with `rustfmt` conventions for clean, consistent output.

Generated test structure:

```rust
#[tokio::test]
async fn test_add_to_cart() {
    let session = startNewSession("chrome").await;

    session.navigate("https://example.com/shop").await.unwrap();
    session.click_intent("Add to Cart").await.unwrap();
    session.fill_intent("Quantity", "2").await.unwrap();
    session.click_intent("Checkout").await.unwrap();
    session.assert_visible("Order Confirmed").await.unwrap();
}
```

#### 6. Visual Assertion Builder

Test authors can pause recording and add explicit assertions by clicking on visible elements:

- **Assert visible** — verify an element is present in the viewport.
- **Assert text** — verify an element's text content matches a value.
- **Assert gone** — verify an element has disappeared.
- **Assert count** — verify exactly N instances of a visual pattern are present.

These are emitted as `session.assert_*()` calls in the generated test.

#### 7. Session Replay for Validation

After recording, the author can replay the generated test within UTO Studio's embedded session to verify it passes before saving. The replay runs the actual `uto run` flow, with real-time visual feedback showing each step resolving against the live app.

### Architecture

#### UTO Studio layers on `uto-ui`

UTO Studio is **not a separate tool**. It is an enhanced mode of the `uto ui` interface:

```
uto-ui/
  src/
    server.rs           — existing HTTP + WebSocket server
    studio/
      recorder.rs       — session event capture (click, type, navigate, swipe)
      inspector.rs      — vision + accessibility overlay protocol
      codegen.rs        — intent-step-to-Rust code generation
      replay.rs         — recorded test replay runner
  assets/
    index.html          — updated SPA with Studio mode toggle
    studio.js           — recording controls, inspector overlay, code preview panel
```

#### Session recording protocol

The recorder wraps an existing `UtoSession` (web or mobile) with a proxy layer that intercepts all method calls and emits `RecordedStep` events:

```rust
pub enum RecordedStep {
    Navigate { url: String },
    ClickIntent { label: String, confidence: f32, bbox: BoundingBox },
    FillIntent { label: String, value: String },
    AssertVisible { label: String },
    AssertText { label: String, expected: String },
    Swipe { direction: SwipeDirection, element_hint: Option<String> },
}
```

Steps are streamed over the WebSocket to the SPA for display and code preview.

#### Code generation

The `codegen` module maps `RecordedStep` sequences to formatted Rust source. It:

- Inlines a test function wrapper from a configurable template.
- Performs deduplication for repeated navigations.
- Adds `// Low confidence — review this step` comments for steps with confidence < 0.7.
- Integrates with the project's `Cargo.toml` to validate that `uto-test` is already a dependency.

#### CLI integration

```
uto ui --studio [OPTIONS]
    --target <web|android|ios>   Target platform for the recording session (default: web)
    --url <url>                  Starting URL for web recording sessions
    --output <path>              File path for generated test code (default: preview only)
```

## Phase 6 Delivery Plan

### Iteration 6.1 — Studio mode scaffold and recorder protocol

1. Add `--studio` flag to `uto ui` CLI command.
2. Implement `RecordedStep` event type and session-wrapper recorder.
3. Update `uto ui` SPA with a basic Studio mode panel (record/stop controls, step list).
4. Stream recorded steps over WebSocket to the SPA in real time.
5. Unit tests for step capture, serialization, and WebSocket relay.

### Iteration 6.2 — Vision inspector overlay

1. Implement the element inspector: on hover, query the vision model for bounding boxes.
2. Stream bounding box + label + confidence data to the SPA overlay layer.
3. Update SPA with visual overlay rendering (SVG bounding boxes over live session screenshot).
4. Add confidence color-coding (green ≥ 0.85, yellow ≥ 0.7, red < 0.7).
5. Integration tests for inspector query/response cycle.

### Iteration 6.3 — Code generation and test file output

1. Implement `codegen` module: `Vec<RecordedStep>` → formatted Rust test function.
2. Add code preview panel to Studio SPA with syntax highlighting.
3. Add "Save to file" action writing generated code to the project's `tests/` directory.
4. Add `rustfmt`-compatible output formatting.
5. Tests for code generation edge cases (low confidence, empty recordings, duplicate steps).

### Iteration 6.4 — Assertion builder and replay validation

1. Implement visual assertion builder (pause recording, click element to add assertion).
2. Add `assert_visible`, `assert_text`, `assert_gone`, `assert_count` assertion types.
3. Implement replay runner: execute generated test steps via `uto run` and stream results back.
4. Add "Validate before save" workflow in the SPA.
5. End-to-end validation on macOS, Linux, and Windows.
6. Add Phase 6 reference project under `examples/phases/phase6-studio/`.

## Done Criteria for Phase 6 MVP

- [ ] `uto ui --studio` launches a recording session for web targets.
- [ ] Live recording captures click, type, and navigation events as `RecordedStep` items.
- [ ] Element inspector overlays bounding boxes and labels during hover.
- [ ] Code generation produces a valid, runnable Rust test function.
- [ ] Generated code is saved to the project's `tests/` directory on request.
- [ ] Assertion builder adds explicit verification steps to the recording.
- [ ] Replay validation runs the generated test and streams pass/fail back to the UI.
- [ ] Mobile recording works for Android targets (Appium).
- [ ] All tests pass on macOS, Linux, and Windows CI.
- [ ] A committed Phase 6 reference project exists under `examples/phases/phase6-studio/`.

## Consequences

### Positive

- UTO becomes a full-stack automation platform: discover, run, debug, **and author** tests.
- Vision-first recording eliminates the selector-fragility problem at the source.
- Cross-platform recording closes the largest remaining capability gap vs. Cypress and Playwright.
- Rust codegen provides a cleaner, more maintainable artifact than JS/TS recorder output.
- Integrated replay validation catches bad recordings before they reach CI.

### Negative

- Recording proxy adds complexity to the `UtoSession` trait interface.
- Vision inspector requires live session screenshot streaming (new capability in `uto-core`).
- Code generation must handle edge cases: multi-platform sessions, low-confidence steps, ambiguous labels.
- SPA grows in complexity; inline HTML/CSS/JS approach may reach ergonomic limits.

### Architectural Implications

- `uto-ui` gains a `studio/` module but remains a presentation and orchestration layer.
- `uto-core/src/session` gains a `RecordingSession` wrapper (non-breaking trait extension).
- `uto-reporter` schema is not modified; `RecordedStep` is a Studio-only type.
- All Phase 1–5 layer boundaries remain intact.
- `uto-cli` gains `--studio` flag on the `ui` command; no new top-level command needed.

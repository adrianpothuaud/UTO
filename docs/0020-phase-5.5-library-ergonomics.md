# ADR 0020: Phase 5.5 — Library Ergonomics and CLI Flexibility

Date: 2026-03-19

## Status

Complete — Phase 5.5 delivered (Library Ergonomics and CLI Flexibility)

## Context

Phase 5bis delivered UI improvements focused on test observation and debugging. Before proceeding to Phase 6 (UTO Studio), two critical usability gaps need resolution:

### Gap 1: UTO Cannot Be Used as a Library in Non-Scaffolded Projects

**Current state:**  
UTO currently expects projects to be initialized via `uto init`, which:
- Creates a specific directory structure (`tests/`, `src/lib.rs`, `uto.json`)
- Generates `Cargo.toml` with path dependencies to `uto-test`, `uto-core`, etc.
- Assumes the user wants CLI-orchestrated test execution

**Problem:**  
Many Rust teams want to integrate UTO into existing projects without:
- Running `uto init` and copying template structure
- Restructuring their codebase to match CLI expectations
- Using `uto run` as the test runner (they prefer `cargo test` directly)

**Expected behavior:**  
```rust
// In any Rust project with uto-test as a crates.io dependency
use uto_test::uto_test;

#[uto_test(target = "web")]
#[tokio::test]
async fn my_existing_test() {
    let session = uto_test::startNewSession("chrome").await.unwrap();
    session.goto("https://example.com").await.unwrap();
    // ... test logic
}
```

This should work without `uto.json`, without `--project`, and without CLI scaffolding.

### Gap 2: UTO CLI Requires `--project` Argument Even When Obvious

**Current state:**  
Every `uto run`, `uto report`, and `uto ui` command requires `--project <path>`:

```sh
cd my-uto-project
uto run --project .       # Redundant when CWD is the project root
uto report --project .    # Obvious intent: report the current project
uto ui --project .        # Same CWD, why specify?
```

**Problem:**  
This is verbose and inconsistent with industry-standard CLI tools:
- `npm test` (no `--project .` needed)
- `cargo test` (no `--project .` needed)
- `pytest` (no `--project .` needed)
- `mvn test` (no `--project .` needed)

**Expected behavior:**  
```sh
cd my-uto-project
uto run              # Infers --project . from CWD
uto report           # Scans CWD for uto.json
uto ui               # Opens UI for current project
```

Explicit `--project <path>` remains available for monorepo scenarios or CI scripts.

---

## Decision

UTO will deliver **Phase 5.5** as a corrective phase to address library ergonomics and CLI usability before Phase 6 (Studio) begins. This phase refactors the CLI and runner architecture to support:

1. **Library-first usage** — `uto-test` works as a standalone crate without CLI scaffolding
2. **Smart CWD inference** — `uto` commands default to the current directory when `--project` is omitted

---

## Phase 5.5 Objectives

### 5.5.1 — Library-First API

**Goal:** Make `uto-test` usable in any Rust project without `uto.json` or CLI scaffolding.

#### Changes Required

| Component | Change |
|---|---|
| `uto-test/src/session.rs` | Make `startNewSession()` and `startNewSessionWithArg()` work without `uto.json` by falling back to sensible defaults |
| `uto-test/src/lib.rs` | Export convenient re-exports: `pub use uto_core::{UtoSession, WebSession, MobileSession};` |
| `uto-core/src/env` | Ensure discovery and provisioning functions are public and callable from library code |
| `uto-core/src/driver` | Ensure driver spawning functions work without project context |
| `uto-test-macros/src/lib.rs` | Make `#[uto_test]` macro work when `uto.json` is absent (optional metadata only) |
| `README.md` + `docs/` | Add "Library Usage" section with non-CLI examples |

#### Workflow

**Without uto.json (library mode):**
```rust
// Cargo.toml
[dependencies]
uto-test = "0.5"
uto-core = "0.5"
tokio = { version = "1", features = ["full"] }

// tests/integration_test.rs
use uto_test::{uto_test, startNewSession};

#[uto_test(target = "web")]
#[tokio::test]
async fn library_usage_example() {
    // Discovers Chrome, provisions ChromeDriver, spawns driver, creates session
    let session = startNewSession("chrome").await.unwrap();
    session.goto("https://example.com").await.unwrap();
    let title = session.title().await.unwrap();
    assert!(!title.is_empty());
    session.close().await.unwrap();
}
```

**Key principle:** `uto-test` should be a zero-config library that "just works" for Rust developers who don't want CLI orchestration.

### 5.5.2 — CWD Inference for CLI Commands

**Goal:** Make `uto run`, `uto report`, `uto ui` infer `--project .` when the argument is omitted and a valid `uto.json` exists in CWD.

#### Changes Required

| Component | Change |
|---|---|
| `uto-cli/src/parsing.rs` | Change `project: String` to `project: Option<String>` in `RunArgs`, `ReportArgs`, `UiArgs` |
| `uto-cli/src/commands.rs` | Add `fn infer_project_from_cwd() -> Option<PathBuf>` that scans CWD and parent directories for `uto.json` |
| `uto-cli/src/commands.rs` | Apply fallback: `let project = args.project.unwrap_or_else(|| infer_project_from_cwd().unwrap_or_else(|| ".".into()));` |
| `uto-cli/src/main.rs` | Update help text: `--project <path>` becomes `[--project <path>]` (optional) |

#### Workflow

```sh
# Before (Phase 4–5bis):
cd my-uto-project
uto run --project .
uto report --project .
uto ui --project .

# After (Phase 5.5):
cd my-uto-project
uto run          # Infers --project . from uto.json in CWD
uto report       # Infers --project . from uto.json in CWD
uto ui           # Infers --project . from uto.json in CWD

# Explicit paths still work (monorepo / CI scenarios):
uto run --project /path/to/other/project
```

#### Discovery Algorithm

```
1. If --project <path> is provided explicitly → use <path>
2. Else, scan CWD for uto.json:
   a. If found → use CWD as project root
   b. Else, scan parent directories up to 3 levels:
      - If uto.json found → use that directory
      - Else → error: "No uto.json found. Run 'uto init' or specify --project <path>"
```

### 5.5.3 — Crates.io Publishing Preparation

**Goal:** Publish `uto-test`, `uto-core`, `uto-reporter`, `uto-logger`, `uto-test-macros` to crates.io so library users can add UTO without local path dependencies.

#### Pre-Publish Checklist

- [ ] All public APIs have Rustdoc comments
- [ ] `README.md` in each crate with usage examples
- [ ] `LICENSE` file in workspace root (MIT or Apache-2.0 dual-license)
- [ ] `Cargo.toml` fields: `description`, `license`, `repository`, `homepage`, `keywords`, `categories`
- [ ] Version bumped to `0.5.0` across all crates
- [ ] CI validates all crates build independently (no workspace-only deps)
- [ ] Integration tests pass with published versions (not path dependencies)

#### Publishing Order

1. `uto-logger` (no dependencies on other UTO crates)
2. `uto-reporter` (depends on `uto-logger`)
3. `uto-core` (depends on `uto-logger`, `uto-reporter`)
4. `uto-test-macros` (depends on none)
5. `uto-test` (depends on `uto-core`, `uto-test-macros`)
6. `uto-runner` (depends on `uto-test`, `uto-reporter`)
7. `uto-ui` (depends on `uto-runner`)
8. `uto-cli` (depends on all above, not published — binary only)

## Implementation: Two-Mode Init Design

### Problem

Originally, `uto init` unconditionally required a UTO workspace root to generate projects with path dependencies. This violated the Phase 5.5 goal of making UTO usable as a library without the source tree.

### Solution

`uto init` now supports two modes:

**Production Mode (default)**
```sh
uto init ./my-project --template web
```
- Generates `Cargo.toml` with crates.io dependencies (`uto-test = "0.5.0"`)
- Works anywhere, no UTO workspace required
- Intended for end users installing UTO via `cargo install uto-cli`

**Development Mode (`--dev` flag)**
```sh
cd /path/to/UTO
uto init ./my-project --template web --dev
```
- Generates `Cargo.toml` with path dependencies (`uto-test = { path = "/path/to/UTO/uto-test" }`)
- Requires UTO workspace root (auto-detected by walking up directory tree)
- Intended for UTO contributors testing CLI changes

### Implementation Details

| Component | Change |
|---|---|
| `uto-cli/src/parsing.rs` | `InitArgs::uto_root` changed from `PathBuf` to `Option<PathBuf>`; added `--dev` flag parsing; auto-detect workspace root only in dev mode |
| `uto-cli/src/templates.rs` | Split `cargo_toml()` into `cargo_toml_crates_io()` and `cargo_toml_path_deps()` |
| `uto-cli/src/commands.rs` | Validate `uto_root` only when `Some(_)`; select template based on mode |
| `uto-cli/src/main.rs` | Updated help text to document `--dev` flag and modes |

### Backward Compatibility

Existing example projects under `examples/phases/` use path dependencies and continue to work. The `--dev` flag preserves this workflow for UTO development.

## Architecture Implications

### What Changes

| Component | Change |
|---|---|
| `uto-test/src/session.rs` | Add optional `uto.json` fallback logic for library mode |
| `uto-test/src/lib.rs` | Export `UtoSession`, `WebSession`, `MobileSession` for library users |
| `uto-cli/src/parsing.rs` | Make `--project` optional in `RunArgs`, `ReportArgs`, `UiArgs` |
| `uto-cli/src/commands.rs` | Add `infer_project_from_cwd()` with 3-level parent scan |
| `uto-cli/src/main.rs` | Update help text to reflect optional `--project` |
| `README.md` | Add "Library Usage" section with non-CLI example |
| `docs/0013-getting-started-and-troubleshooting.md` | Add library usage troubleshooting |

### What Does Not Change

- `uto-core` architecture (env/driver/session separation)
- `uto-reporter` schemas (`uto-report/v1`, `uto-suite/v1`)
- `uto-ui` server or SPA implementation
- CLI command names or top-level workflows
- Phase 1–5bis delivery contracts
